## 1. 项目概述

### 1.1 项目背景

建设一个以博客为核心的个人网站，承载文章发布、用户互动等基础功能，同时预留扩展能力，未来可作为个人应用的统一入口（应用门户）。

### 1.2 项目目标

- 搭建一个可独立部署的个人网站
- 实现博客文章发布、展示、互动（评论/点赞）的完整闭环
- 建立用户与权限体系，区分游客、注册用户、管理员
- 架构上预留应用门户能力，后续可接入其他自部署应用

### 2.5 应用门户模块（核心扩展）

这是你描述的核心扩展场景：将个人部署的其他应用集成到网站中，作为统一入口。

#### 2.5.1 功能描述

> **场景举例**：你开发了一个数据分析工具，部署在服务器的 `localhost:8080`，你希望在个人网站的导航栏有一个"数据分析"入口，有权限的用户点击后可以直接访问该应用。

#### 2.5.2 需求拆解

| 编号 | 需求描述 | 优先级 |
|------|---------|--------|
| F-46 | 管理员可在后台添加"应用入口"，配置：应用名称、描述、图标、目标 URL（如 `http://localhost:8080` 或 `/app/data-tool`） | P1 |
| F-47 | 每个应用入口可独立设置访问权限（游客可见 / 仅登录用户 / 仅管理员） | P1 |
| F-48 | 前端导航栏或独立"应用"页面展示当前用户有权限访问的应用入口 | P1 |
| F-49 | 点击入口后跳转方式可选：新标签页打开 / 当前页内嵌 iframe | P2 |

#### 2.5.3 技术方案（已确定：反向代理）

采用 **Nginx 反向代理** 方案，所有应用通过统一域名访问：

```
用户浏览器
    │
    ▼
https://你的域名.com/app/tool/   ← 用户始终只看到你的域名
    │
    ▼ (Nginx 在服务器内部转发，携带用户认证信息)
http://127.0.0.1:8080            ← 内部应用，不直接暴露在公网
```

**工作流程**：
1. 管理员在后台添加应用入口，配置名称和代理路径（如 `/app/tool`）及内部目标地址（如 `http://127.0.0.1:8080`）
2. 系统自动生成 Nginx 反向代理配置（或手动添加）
3. 用户访问 `/app/tool` 时，Nginx 将请求转发到内部端口，同时携带用户认证信息
4. Nginx 层校验登录状态：未登录用户访问受保护的应用时，重定向到登录页

#### 2.5.4 安全考量

- Nginx 代理层校验用户身份（通过 Cookie/Token），拦截未授权请求
- 内部应用监听 `127.0.0.1` 而非 `0.0.0.0`，端口绝不直接暴露在公网
- 考虑 CSRF 防护，尤其 iframe 内嵌场景
- 内部应用间的请求隔离，避免互相访问

## 4. 技术选型（已确定）

| 层级 | 选型 | 选择理由 |
|------|------|---------|
| **语言** | TypeScript | 前后端类型统一，AI SDK 生态活跃，流式响应处理自然 |
| **后端框架** | Node.js + Fastify | 轻量高性能，TypeScript 一等支持，插件化易扩展 |
| **数据库** | SQLite | 个人博客零维护，后续可无缝迁 PostgreSQL |
| **ORM** | Drizzle | 类型安全，SQL-like API，轻量无黑盒 |
| **前端框架** | Next.js | SSR/SSG 兼顾 SEO 与性能，API Routes 可复用后端逻辑，AI 交互友好 |
| **Markdown 渲染** | unified + remark + rehype | 插件生态丰富，支持语法高亮、数学公式、TOC 生成 |
| **样式** | Tailwind CSS | 原子化 CSS，响应式方便，暗色模式支持好 |
| **部署** | Docker + Nginx + Let's Encrypt | 一键部署，HTTPS 自动续签 |
| **图片存储** | 本地文件系统 | 初期够用，后续可迁 S3/R2 |

### 为什么选 TypeScript 全栈

```
前端 TypeScript  ←——类型共享——→  后端 TypeScript
      │                                  │
      │  AI SDK (Vercel AI SDK,          │
      │  Anthropic SDK, OpenAI SDK)      │
      │      一等公民支持                 │
      └──────────────┬───────────────────┘
                     │
                AI 功能扩展
        （自动摘要、智能搜索、写作助手等）
```

- **AI 接入优势**：TypeScript 是 AI SDK 的第二大支持语言（仅次于 Python），流式响应、Agent/Tool Use 等能力社区最活跃
- **维护成本**：一套类型定义前后端共用，改接口不会漏
- **性能**：Fastify 是 Node 生态最快的 HTTP 框架，个人博客绰绰有余

## 5. 数据模型概要

### 5.1 核心实体

```
User（用户）
  ├─ id, email, password_hash, nickname, avatar, bio
  ├─ role: guest(隐式) / user / admin
  ├─ status: active / disabled
  └─ created_at, updated_at

Article（文章）
  ├─ id, title, slug, summary, content(markdown), cover_image
  ├─ status: draft / published
  ├─ author_id → User
  ├─ tags (多对多)
  ├─ published_at, created_at, updated_at
  └─ view_count, like_count(冗余缓存)

Comment（评论）
  ├─ id, content, article_id → Article
  ├─ author_id → User
  ├─ parent_id → Comment (自引用，支持嵌套回复)
  └─ created_at

Like（点赞）
  ├─ id, article_id → Article
  ├─ user_id → User
  └─ created_at
  └─ 唯一约束: (article_id, user_id)

AppEntry（应用入口）
  ├─ id, name, description, icon
  ├─ proxy_path: 对外路径（如 /app/tool），用户通过 https://域名.com/app/tool 访问
  ├─ target_url: 内部目标地址（如 http://127.0.0.1:8080），Nginx 将请求转发至此
  ├─ open_mode: open / login_required / admin_only
  ├─ sort_order
  └─ created_at

SiteConfig（站点配置）
  ├─ key, value (键值对)
  └─ 示例: site_name, site_description, logo_url
```

### 5.2 关系概览

```
User ──1:N──→ Article
User ──1:N──→ Comment
User ──1:N──→ Like
Article ──1:N──→ Comment
Article ──1:N──→ Like
Article ──N:N──→ Tag
```
